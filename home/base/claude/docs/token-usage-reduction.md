# Claude Code Token-Usage Reduction

## Goal and fixed constraints

This document ranks the levers that cut weekly token consumption on a Max 20x subscription, where the weekly rate limit is the real budget. The dominant cost is conversation-history re-read (cache_read), not the static prefix. Every lever below is grounded in the real repo config and, where it came from web research, cites the source URL.

These are deliberately off-limits. Do not propose touching them:

- Model stays `claude-opus-4-8` (`home/base/claude/settings/global-settings.nix:25`). No downgrade to Sonnet/Haiku for interactive or workflow work.
- Reasoning effort stays `max` (`effortLevel="max"`, `global-settings.nix:26`).
- `ultracode=true` and `enableWorkflows=true` stay (`global-settings.nix:27-28`). The workflow/subagent fan-out pattern is required for quality and is itself a token saver (see lever 2).

Compaction mechanics (the 200K clamp, the trigger arithmetic, the resume-500 path, the 1M window) are documented in `home/base/claude/docs/context-management.md`. This doc cites that file and goes beyond it; it does not restate the arithmetic.

## Measured baseline

Kira host, `~/.claude` transcripts, 14 days, all Opus 4.8:

- 16.2B total tokens. 92% cache_read, 7% cache_write, 0.4% raw input, 108M output (0.7%).
- cache_read per request: median 81.5k, p90 420k, max 999k. This is the conversation history re-read every turn. It is the dominant cost.
- Static prefix (system + tools + CLAUDE.md + skills) first cache_write per session: median 15.4k, p90 24.7k. Prefix trimming is a minor lever.
- The three CLAUDE.md/core.md files total ~5.6k tokens. Trimming them is noise; adding ~300 tokens to them is also noise.
- Cache hit ratio 92.5%; cache_write low. Caching is already efficient. The job is to not regress it and to shrink history, not to add breakpoints.
- Autocompact fires at ~900k tokens (`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=90` against the native 1M window).
- Attribution: heavy interactive coding sessions dominate (triage 31%, ai-first 25%, dotfiles 15%, orchestrator 14%); autonomous agents are minor (steward 4.4%, jenny 0.1%).
- 72% of usage happened while 4+ sessions ran in parallel against the shared weekly limit. 25% of usage ran at >150k context.

The structural fact behind every lever: each tool result and each assistant message is appended to the `.jsonl` history and re-sent as cache_read on every subsequent turn until compaction or `/clear`. A byte added once is re-billed N times. So the highest-impact levers either keep bytes out of history (output caps, read discipline, subagent absorption) or reset history sooner (`/clear`, fewer parallel sessions, earlier compaction).

## Ranked levers

Impact reflects effect on total weekly token volume given the baseline. "Ready" means it can ship now under the repo workflow (edit, format, commit specific-file, rebuild, `tests/run.sh`); "needs sign-off" means it changes interactive UX or behavior the user should approve first.

### Theme A: parallel-session and history lifecycle (the 92% directly)

**1. Cap concurrent heavy Opus sessions at 2-3, not 4+. Impact: high. Needs UX sign-off.**
Mechanism: cache_read is paid per session per turn. N parallel heavy sessions multiply the dominant cost ~linearly. 72% of usage ran at 4+ parallel against one shared weekly limit; a 4th idle-but-open session at p90 420k history costs ~420k cache_read on its next turn for zero throughput. Apply: policy in `interactive-preferences.md` to run at most 2-3 heavy interactive Opus sessions concurrently and serialize the rest; keep autonomous agents tiny (already true); surface a live count of sessions >100k context so crossing 3 is visible (pairs with lever 11). Workflow subagents are isolated child contexts, not heavy parallel human sessions, so they do not count. Risk: lower wall-clock parallelism on independent streams; serialized sessions finish faster with less context rot. This is the explicit budget trade. Estimate: dropping the 72%-of-usage 4+-parallel windows toward 2 parallel roughly halves live histories in those windows, plausibly 25-35% off total weekly volume. This is the single largest lever and it touches no model/effort/workflow knob.

**2. Route heavy exploration through subagents/workflows as a deliberate token saver. Impact: high. Ready (instruction edit).**
Mechanism: a subagent runs in its own context window; its file reads, Grep results, and exploratory tool calls stay in the subagent and never enter the parent history, so they are never re-billed as parent cache_read. Only the final summary returns. A researcher that reads 300k of files returns ~2-5k of synthesis. `context-management.md:31` mentions subagents for window exhaustion but not for per-turn re-read-base reduction. Apply: add one clause to the `<delegation>` block in `agents/core_rules/core.md:43` framing delegation of heavy/multi-file reads as a token lever (author via the `instructions` skill, machine-consumer surface). Reinforce in `agents/skills/research/SKILL.md` and `agents/skills/deliver/SKILL.md` that subagents must return compact synthesis, never paste file bodies (`research` already returns TL;DR-only). Source: https://code.claude.com/docs/en/sub-agents. Risk: a subagent spawn pays a fresh ~15.4k prefix write, so scope this to multi-file or whole-file-survey reads, not single reads. ultracode/workflows are mandated to stay, so this only intentionalizes the required pattern. Estimate: one in-thread 300k investigation re-billed across a 20-turn session can cost millions of cache_read tokens; absorbed in a subagent it costs ~300k once plus ~15k prefix plus ~3k summary. Behavioral, bounded by adoption.

**3. `/clear` (or fresh session) at every task/commit boundary. Impact: high. Needs UX sign-off.**
Mechanism: a session that drifts across three tasks re-reads the cumulative history of all three on every later turn. `/clear` wipes the in-session conversation so the next turn re-reads only the ~15.4k prefix plus the new task. The best clear point is right after a commit: work is saved, nothing needs carrying. CLAUDE.md/skill/memory files survive `/clear` and are re-read fresh, so durable knowledge is not lost. Apply: rule in `interactive-preferences.md`/CLAUDE.md to `/clear` after a task-closing commit before an unrelated task; trigger on correcting Claude >2x on the same point (cluttered failed-approach context is pure overhead); prefer a new session over `/resume` of a bloated one (also sidesteps the resume-500 path at `context-management.md:5-11`); optional Stop-hook advisory when a commit just landed and context >100k or >~50% with a topic change (advisory only, never auto-clear). Risk: clearing mid-task loses needed context, so fire only at true boundaries. Estimate: a 400k-history session reset to a 15k prefix before an unrelated task removes ~385k cache_read on the next and every following turn. For the 25%-of-usage >150k cohort, disciplined per-task clearing could pull working context toward 60-90k, cutting cache_read ~40-60% on those turns. Plausibly double-digit-percent of cache_read overall.

### Theme B: tool-output and MCP-output capping (keep bytes out of history)

**4. Set `MAX_MCP_OUTPUT_TOKENS` below the 25k default. Impact: high on browser-heavy sessions, medium overall. Ready.**
Mechanism: MCP results default to a 25,000-token cap with a warning already shown at 10,000 (https://code.claude.com/docs/en/mcp). The always-connected devtools targets (`chrome-devtools` and, on chise, `vivaldi-devtools`, where `take_snapshot` returns the full a11y tree and `take_screenshot` returns image data) are the only uncapped high-volume emitters left after Read self-bounds. Each such result can land up to 25k tokens permanently, then re-read at the 81.5k-420k/turn rate. Apply: add `MAX_MCP_OUTPUT_TOKENS = "10000";` to `environment-variables.nix` (unset today, confirmed by grep). Note the conflict: researchers split between `"10000"` (matches the warn threshold, three of seven) and `"25000"` (conservative headroom, two of seven). Recommend `"10000"`: it is a 60% cut on each browser/devtools dump and the saving multiplies by turn count; a server that declares `anthropic/maxResultSizeChars` keeps its own limit regardless, and `codex` summaries are naturally small, so the cap bites exactly the bloated browser surfaces. Risk: a truncated DOM/snapshot may force a re-scope call, which is cheaper than carrying a 25k blob every turn. Mitigate behaviorally with lever 6. Source: https://code.claude.com/docs/en/mcp. Estimate: episodic but high-magnitude; a single uncapped 40k DOM re-read across ~20 turns is ~800k cache_read from one call. Modest on pure-coding sessions.

**5. Set `BASH_MAX_OUTPUT_LENGTH` to spill large Bash stdout to a file. Impact: medium. Ready.**
Mechanism: when a Bash result exceeds this character count, Claude Code saves the full output to a file and the model receives the path plus a short preview instead of the whole stream becoming permanent history. Default is 30,000 chars (~8k tokens) with middle-truncation (anthropics/claude-code issues #19901, #40100). Apply: add `BASH_MAX_OUTPUT_LENGTH = "16000";` to `environment-variables.nix` (unset today; only the BASH timeouts are set). 16000 keeps head+tail context via middle-truncation, sits above a normal test-suite tail, and routes anything bigger to a spill file. Do not go below ~8000 or legitimate stack traces clip. Risk: low; the full output is on disk and re-readable on demand, and head/tail (where errors and exit codes live) are preserved. Source: https://code.claude.com/docs/en/env-vars, issue #17611 (Bash/Read/Grep/Glob measured at 20-30% of token overhead in automation-heavy sessions). Estimate: most Bash calls are already under 30k, so this is a tail-risk cap; honestly medium.

**6. Behavioral cap on screenshot/visual MCP usage. Impact: medium. Ready (skill edit).**
Mechanism: on Opus 4.8 a screenshot costs up to 4,784 visual tokens (the Opus 4.7+ visual budget, ~3x the 1,568 of older models) and stays in history every turn; `MAX_MCP_OUTPUT_TOKENS` caps image content too but a capped image is still a large permanent block. The cheapest fix is to not emit images when a text a11y snapshot answers the question. `take_snapshot` is the chrome-devtools-recommended default over `take_screenshot`, and `take_screenshot` defaults `includeSnapshot=false` so a screenshot does not also drag the snapshot. Apply: add a directive to the `browser`/`pinchtab` skills (authored via the `instructions` skill, not core.md, to avoid bloating the prefix): prefer `take_snapshot` over `take_screenshot`; never request full-page screenshots or `browser_get_html` when a scoped `browser_extract_content` query answers the question; screenshot only for genuine visual verification. Pairs with the hard cap in lever 4 (a skill instruction can be ignored under load). Sources: https://docs.anthropic.com/en/docs/build-with-claude/vision, https://github.com/ChromeDevTools/chrome-devtools-mcp/blob/main/docs/tool-reference.md. Risk: low; text snapshots are the documented primary tool. Estimate: each avoided full-page screenshot saves up to ~4.8k visual tokens permanently; zero on pure-coding turns.

### Theme C: read discipline (the create-side of the 92%)

**7. Read-size discipline in the always-on core.md `<tools>` block. Impact: high. Ready (instruction edit).**
Mechanism: a 2000-line whole-file Read (~20-25k tokens, near Read's 25k self-limit) read once at turn N is re-billed every later turn. The `<tools>` block at `core.md:31` dictates which tool but never the size: no line-range guidance, no Grep-first-then-Read, no don't-re-read. Apply: extend `core.md:31` with ~6 lines (machine-consumer surface, author via `instructions` skill): Grep to locate then Read with offset/limit around the hit, not the whole file; never Read a file already in context this session, reference it by `path:line`; for Bash output over ~100 lines redirect to a file and Grep it; prefer Grep `-n` with `-A`/`-B` windows over Read when only a region is needed. Keep it under ~6 lines to stay inside the ~5.6k always-on budget. Risk: over-narrow reads can miss surrounding context and cause a wrong edit then a re-Read, netting more tokens. Scope the rule to "narrow when you know the target," not "never read whole files." Estimate: if careless whole-file reads contribute 10-20k to a typical 81.5k re-read base and narrow reads cut that region to 2-4k, the per-turn base drops ~8-18k, compounded across long sessions and the dominant coding repos; plausibly 8-15% off cache_read, depending on current habits.

**8. PreToolUse Read-size advisory hook (mirror the write-side guard). Impact: medium. Ready (new hook).**
Mechanism: the existing `line-count-advisory-guard.py` fires only on Write/Edit (`tool-event-registrations.nix:100`); there is no Read-side equivalent, yet Read is where re-read weight is created. Apply: add `agents/hooks/pre-tool-use/read-size-advisory/read-size-advisory.py` registered under a `matcher = "Read"` block; if Read targets an existing file whose line count exceeds the existing threshold (reuse `line_count_policy.count_lines_in_file` / thresholds from `agents/hooks/post-tool-use/line-count/line_count_policy.py`) and `tool_input` has no `limit` key, emit `hookSpecificOutput.additionalContext` advising Grep-then-offset/limit. Advisory only, never blocking; mirror the exit-0 silent-pass of the write-side guard. Risk: advisory noise adds its own small per-call context and can train ignore-fatigue (like memory-recall); keep the threshold high (only above the 200-line blocking threshold), debounce per-session like `memory_recall_debounce.py`, keep logic to a stat plus line count. Estimate: catches the read-2000-lines-when-40-needed lapse (~25k vs ~600 tokens, re-billed every later turn). Backstop to lever 7, not the primary fix.

### Theme D: hook-injection trimming (memory-recall is the only history-additive hook)

Measured: `memory-recall.py` runs on every PreToolUse (`.*`) and injects `Recall: @path` lines. Each event persists ~118 tokens (the harness stores both the raw stdout JSON and a parsed content copy) permanently. A 2445-line session (`603dd116`) had 292 recall events totaling ~35k tokens of permanent recall text. The current debounce gates only on time (30s window, 15s hard floor) and keyword overlap (0.5), never on the emitted path set, so the same files re-inject as keywords drift.

**9. Make memory-recall edge-triggered on the recall path SET. Impact: high (on long sessions). Ready (hook edit).**
Mechanism: the debounce re-injects the same path list whenever keywords drift, so identical sets land many times, each copy permanent. Apply: in `memory-recall.py:86-91` after `select_top_recall_paths`, hash the sorted recall-path set and keep a session-persistent set of already-injected hashes in the `/tmp` state file; if the new set (or a subset) was already injected this session, `exit_silently()`. Risk: low; the earlier identical Recall line is already in history, so re-injecting is pure duplication. Estimate: in 200-300-event sessions drawn from a 24-file memory pool, distinct sets are far fewer than events; deduping plausibly removes 50-70% of events, ~15-25k tokens on the longest sessions.

**10. Cap memory-recall accumulation per session (hard budget). Impact: high (on long sessions). Ready (hook edit).**
Mechanism: nothing today bounds total recall injections, so they grow unbounded with session length. Apply: in `memory_recall_debounce.py` persist a running counter alongside the timestamp/keywords; add `SESSION_RECALL_EVENT_BUDGET` (~40) and a char budget (~20000); in `memory-recall.py` after loading state, `exit_silently()` once the budget is hit; increment in persist. Caps permanent recall at ~5k tokens/session regardless of length. Risk: low; the agent loses new associative recalls late in a session, but memory files stay reachable via `@path`/Read and the most relevant recalls fire early. Estimate: caps the ~35k worst case to ~5k, a ~30k reduction on the longest sessions. Pairs with lever 9.

**Sub-levers (lower priority, ready):** shorten injected recall path strings by stripping the repeated `projects/-Users-lucas-zanoni--dotfiles/memory/` prefix to `@memory/<stem>.md` in `memory_recall_io.py` (saves ~40-50% of each event's path bytes, both stored copies); widen the debounce (`DEBOUNCE_SECONDS`->90, floor->45, overlap->0.35) for a modest further cut. Shorten the `tldr-reminder.py` reminder from ~984 chars to a ~250-char pointer since the Stop hook (`end-of-turn-format-guard`) enforces the same rules reactively; this fires on every keyboard-interactive user prompt (~246 tokens, 2-12x/session). **Do not touch the guards or `workspace-directory-injector`:** they inject nothing on the happy path (`prohibited-command-guard.py:110-111`, `prohibited-words-guard.py:163-164` both `sys.exit(0)` silently), so trimming them loses safety for zero token gain.

**12. Scope dotfiles in-repo skill discovery so a dotfiles session does not load all 37 SKILL.md dirs. Impact: medium (on dotfiles sessions only). Needs verification before sign-off.**

Estimate: ~3.7k tokens off the prefix of dotfiles sessions, re-read every turn. The largest concrete prefix reduction available, still modest against the 81.5k median cache_read.

**13. Disable rarely-used MCP servers at the user scope, re-enable per-agent. Impact: low. Partly done.**

### Theme F: caching mechanics (already optimal; these are guards)

**15. Never set `FORCE_PROMPT_CACHING_5M`; leave `ENABLE_PROMPT_CACHING_1H` unset. Impact: low (guard). Ready (no-op + doc).**
Mechanism: on a Claude subscription Claude Code auto-requests the 1-hour extended cache TTL at zero extra cost; the obvious lever is already on and free. `ENABLE_PROMPT_CACHING_1H` is API-key/Bedrock/Vertex-only and a no-op here. `FORCE_PROMPT_CACHING_5M` would shorten the TTL and expire warm prefixes faster, cutting the 92.5% hit ratio. Grep confirms no `PROMPT_CACHING` var anywhere in the repo (defaults intact). Apply: leave it that way; this doc is the guard. Source: https://code.claude.com/docs/en/prompt-caching. Estimate: 0% additional savings; pure regression insurance.

**16. Avoid mid-session model/effort churn (cache is keyed by both). Impact: medium (preventive). Ready (policy).**
Mechanism: the cache key includes model and effort, each with its own cache; a `/model` or `/effort` switch mid-session reprocesses the entire history with no cache hits (re-pays the full p90 420k as input). Effort stays max and model stays opus-4-8, so the concern is purely not toggling them and not using the `opusplan` alias (which switches Opus<->Sonnet on every plan-mode toggle). Apply: confirm `global-settings.nix:25-26` pins model and effort (it does) and `opusplan` is unused; note in interactive guidance to spawn a subagent for a cheaper sub-task rather than switching the main loop. Source: https://code.claude.com/docs/en/prompt-caching. Estimate: each avoided switch saves one full-history re-read; preventive, bounded by how often switches would otherwise happen.

**17. Keep MCP tools deferred so server connect/disconnect never invalidates the prefix. Impact: medium (preventive). Ready (audit).**
Mechanism: when tools are deferred, a server connecting/disconnecting mid-session only appends and leaves the cached prefix intact; an `alwaysLoad` exemption or forcing tools upfront would re-expose the whole history to every reconnect. The harness already defers on Opus 4.8. Apply: never add an `alwaysLoad` exemption or a config that pulls MCP defs into the prefix; do not set `--strict-mcp-config` in a way that forces tools upfront. Optionally pin `ENABLE_TOOL_SEARCH="true"` to harden against the documented fallback that loads all schemas upfront if `ANTHROPIC_BASE_URL` ever points to a non-first-party host or Vertex (0 tokens today, prevents a ~3-6k/turn prefix regression). Source: https://code.claude.com/docs/en/prompt-caching, https://code.claude.com/docs/en/mcp.

### Theme G: compaction and clawde cadence (already-deployed-adjacent, mostly low)

**18. Optionally lower the autocompact trigger from ~900k. Impact: medium-to-high. Needs quality sign-off.**
Mechanism: turns spent above the trigger re-read a growing raw history; compacting sooner shrinks the re-read base earlier. The deployed config now compacts at ~900k (1M window × 90%), so the raw history is allowed to grow very large before the first lossy summary, and every turn above the trigger re-reads it. Apply (within the 1M window): lower `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` and keep `CLAUDE_CODE_AUTO_COMPACT_WINDOW=1000000`, e.g. `60` gives `min(1000000x0.60, 987000) = 600000`. Apply (back to the old quality-first regime): re-add `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` and set the window+percentage to a 200K-clamped trigger such as `200000` × `45` = 90k. Document the chosen arithmetic alongside the derivation in `context-management.md:21`. Note the tension: the deployed config deliberately uses the full 1M window and accepts late, lossy compaction in exchange for raw context capacity, so this lever trades that capacity (and, near the top of the window, output quality) back for tokens; ship it only if the weekly budget is actually binding. Risk: compaction is lossy. One-line reversible edit; validate subjectively before committing. Estimate: medium-to-high because the current ~900k trigger leaves a large re-read base on long sessions; a 600k (or 200K-clamped) trigger caps it materially, bounded by how often sessions actually climb that high.

**19. Add a CLAUDE.md compact preserve-instructions block. Impact: low (enabler). Ready.**
Mechanism: does not cut tokens itself; it is the quality guardrail that makes levers 3 and 18 safe. Claude Code honors a `## Compact Instructions` block: named items (modified file paths, exact test/rebuild commands and pass/fail, errors and fixes, architectural decisions with rationale) are preserved verbatim in the summary while everything else is compressed harder, preventing re-derivation cost. Apply: add the section to `core.md`/the dotfiles CLAUDE.md (author via `instructions` skill); it survives `/clear` and `/compact`. Estimate: ~0 direct reduction; ~300 added prefix tokens (noise); unblocks the higher-impact compaction/clear levers.

**20. Leave clawde cadences as-is. Impact: low (guard). No action.**
Mechanism: steward fires every 15 min (change-gated), jenny every 30 min (ungated); both intervals sit inside the 1-hour TTL so the prefix stays warm between ticks, and `daily_session_rotation=true` forces one fresh prefix/day. Shortening a heartbeat to "keep cache warmer" burns more real turns than the prefix it protects. Steward is 4.4% and jenny 0.1% of usage. Apply: keep intervals between 30 and 60 min; honor stewardship by changing any gate/driver behavior in the clawde flake, not by hand on the running loop. Estimate: ~0 recoverable; flagged to prevent a counterproductive tighten-heartbeat change.

## Out of scope (rejected)

These would help but violate the fixed constraints or have no supported surface, recorded so the tradeoff is explicit:

- **Downgrade the model to Sonnet/Haiku for cheap sub-tasks or per-tool routing.** Would cut per-token weight but violates the model=opus-4-8 constraint and the quality bar. Per-tool model routing (issues #19269, #15721) is also not shipped.
- **Lower reasoning effort below max.** Effort=max is fixed; `MAX_THINKING_TOKENS` is the effort knob and is off-limits.
- **Disable ultracode or reduce workflow/subagent fan-out.** Mandated to stay; subagent isolation is itself a token saver (lever 2), so reducing it would raise cost.
- **Set `MAX_OUTPUT_TOKENS` higher to read large files in one call.** Would let multi-hundred-k-token files into permanent history and spike the dominant cache_read metric. Keep it unset; this is a do-not-regress guard, not a lever.
- **Set `CLAUDE_CODE_MAX_OUTPUT_TOKENS` to clip Claude's own output.** Output is 0.7% of volume; the var is reportedly ignored/hard-capped at 32k on Opus 4.6+ (issues #24159, #25569) and not applied to subagent Task calls. Clipping deliverables is a quality hit for negligible savings. The `<audience>` no-paste rule already covers the safe case.
- **Inject the API context-editing beta (`clear_tool_uses_20250919`) via `CLAUDE_CODE_EXTRA_BODY`.** Sounds high-impact (the headline 84%/100-turn figure) but there is no supported Claude Code setting; the harness already runs this machinery internally as microcompact + Tier-2 server-side clearing. Manual injection could collide with built-in microcompact and invalidate the warm prefix, raising cache_write net-negative. Tracked by issue #26215; do not deploy without throwaway-session measurement.

- **Force the 1-hour cache TTL / pre-warm the prefix.** The 1h TTL is already on and free on subscription; pre-warming (`max_tokens:0`) is net-neutral on tokens (a latency play only).
