Memory in AI agents is not an abstract capability, it is instruction wiring that forces the context window to ingest a specific file at a specific moment, and the entire design problem reduces to deciding which file gets injected, when, and through which trigger, whether that is an @-file reference typed by the user, a global CLAUDE.md loaded at session start, or a routine the agent is taught to run that reads other files on demand. Wiring alone is insufficient because dropping bytes into context does not make the model act on them, so every injection has to ship paired with behavioral instructions that establish authority level, verification rules, and write semantics, otherwise the memory is just noise the model is free to ignore. The cost model is brutal and always-on injection compounds linearly with every turn, which means a memory that lives permanently in context pays for itself on every forward pass forever, and since a small context is one of the dominant factors in agent assertiveness, the architecture has to tier the budget into always-on for identity-level facts, session-start scan for active work, and on-demand for everything else, mirroring how skill descriptions are loaded eagerly while skill bodies are loaded only when invoked. On top of that pipeline a precise imperative workflow has to be defined that tells the agent what the memory is for, when to reach for it, and how to retrieve it, because without an explicit discovery protocol the agent cannot recognize that a given task requires memory in the first place, and the entire mechanism collapses into either silent ignorance or wasteful blanket loading. Memory therefore is not a feature, it is a planned injection pipeline plus a discovery procedure plus a behavioral contract, and any one of those three missing makes the other two useless.
</memory>
